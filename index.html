What is a Technical Design document?

The Technical Design document is the blueprint that all the engineers on the team will need to understand and use on the process of the creation of a game, the TDD needs to explain not only what will be created it also explains how it will be implemented.

There is a common misunderstanding of what is an TDD, in game development it also exist the GDD (game design document) the definition of both look similar, but the crucial difference is that the TDD it specifies more technically the different parts of the creation of the game and the GDD only describes the game itself and not how it is done. There is also another TDD (test driven development) but this one has nothing to do with the Technical design document.

Purpose of the technical design document:

Besides the developers of the team the TDD will also be used to explain the game to any publishers our vision and use it as an early milestone to the publisher, the problem that appears with this is that the developers are pressured on the early stages and there is not much time to think the TDD, because the publisher normally prefer seeing something working on the screen, in the stage of the TDD creation is really important because detecting a defect on an early stage like this will save lots of time, energy and money to the team, so the more time expended making a good TDD the less problems we will be facing in the future.

Which is the moment to make the TDD?

Normally the TDD needs to be created on the preproduction stage alongside with the document design , during the stage of production there are times when it is necessary to change the TDD , there is nothing wrong on adapting de TDD in response to any change in the game.

What really goes inside a TDD?

The TDD we actually can see it as shopping list of the requirements of the game, which we define whom, when and how this conditions will be achieved, we can see the TDD as a miniature of the project with several stages, in this stages there will be many diagrams and time estimates to complete the different tasks.

Requirements Capture:

Requirements capture is the part when we identify all the requirements the game has to satisfy, in this process everyone in the team takes part writing down their expectations, and it can be anything from going to 60 fps to having 3d sound and much more like this, but this stage will be the most critical to a successful project and is also the most difficult. It is difficult to know when you have enough requirements or if you have too many of them, in this stage it will be helpful using the UML system to express ideas.

Code style:

In the TDD is also specified the code style, this way is easier for the developers to understand between themselves and the final code is much more clean, for example everyone who codes should follow the same rules when coding, like:

- All code should be in english
- Spacing equally when declaring a function
- Declaring variables
- Commenting code
- Naming
- Libraries hierarchy

Plataform:

The TDD also specifies the main platform the game will be made for, in the case for pc at this moment there are so many different that it will specify the minimum requirements to run.

For example:

Performance budget:

This section of the TDD will be used to describe the goal for the performance of the game running in the desired platform, for example running at least at 60 fps on pc.

Branch workflow:

The explanation of all the branches from the project, the importance of each one and a short definition of the reason of its use, for example:
A project with 3 different branches
- Master: A branch only for releases of the game.
- Development: A branch to progress with the game.
- Bugs: A branch for bug fixing.

Software and hardware needed:

The TDD will also have a list with all the different software that the team will need and also the hardware for example:
Software: Photoshop, Tiled, Github, 3DS Max, Visual studio.
Hardware: The pcs used and their specifications

Code reviews:

Inside the TDD there’s also need to be a dedicated part for the code review, this way it will be much easier for developers to rank any kind of bug, with this bug report template for all the team when someone finds out a bug it will also have to write accurate report about it.

Code organization overview (UML):

Inside the TDD there is also one UML that will represent the main structure of the code, there is no problem if this structure has to evolve in the future to adapt to the development process.
